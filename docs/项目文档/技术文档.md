## 核心架构决策

### 为什么用 DeepSeek 而非直接调 OpenAI

DeepSeek API 完全兼容 OpenAI SDK 接口，只需修改 `baseURL`。性价比更高，中文能力优秀。切换其他兼容提供商只需改环境变量。

### 为什么用 Better Auth 而非 NextAuth

Better Auth 对 email/password 登录有更简洁的 API，与 Drizzle ORM 集成更好，框架无关的设计在 Next.js serverless 环境下更稳定。

### 为什么数据库在本地而非云上

项目初期降低运营成本。通过域名 `db_qfriendly.qfxblacker.top` 反向代理到本地 PostgreSQL，Vercel Functions 通过公网访问。适合低流量阶段，后续可迁移到 Neon 等云数据库。

## 认证流程

```
用户提交 email/password
    ↓
Better Auth API (/api/auth/[...all])
    ↓
drizzleAdapter → PostgreSQL (user/account/session 表)
    ↓
Set-Cookie: better-auth.session_token
    ↓
后续请求: auth.api.getSession(headers) 校验
```

管理员识别: 查询 `subscriptions` 表中 `variant_name = 'admin' AND status = 'active'`。

## AI 对话架构

### 对话流程

```
用户发消息 (POST /api/chat)
    ↓
订阅门控检查 (checkSubscriptionGate)
    ↓ trial → 订阅 → 专家锁定 → 日限额
    ↓
创建/查找 conversation → 存储 user message
    ↓
加载历史消息 (最近 20 条)
    ↓
构建 chatMessages = [System Prompt, ...history, user message]
    ↓
DeepSeek API stream (model: deepseek-chat)
    ↓
SSE 推送到前端 → 存储 assistant message
```

### System Prompt 设计

四位专家各有独立的中英文 System Prompt（`lib/prompts/experts.ts`）。每位专家有明确的人格、擅长领域、语调和行为准则。语言根据请求参数动态选择。

### 专家切换

通过 `/api/chat/switch` 实现。无历史消息时返回固定欢迎语；有历史时构建包含前 10 条消息的过渡 prompt，由 AI 生成自然过渡。

## 订阅门控

```
POST /api/chat
    ↓
checkSubscriptionGate(userId, expert, 'consume')
    ↓
1. 查询 subscriptions 表（按 userId）
    ↓
2. 未订阅 → checkTrialAccess (SELECT FOR UPDATE 原子递增 trial_used)
    ├─ trial_used < 3 → 放行
    └─ trial_used ≥ 3 → 402 TRIAL_EXHAUSTED
    ↓
3. 已订阅 → 专家锁定检查
    ├─ variant = 'start' + expert ∈ {noah, adrian} → 403 EXPERT_LOCKED
    └─ 其他 → 继续
    ↓
4. 日限额检查（所有有限额方案）
    查询当日 messages 中 role='user' 的数量
    ├─ 超过限制 → 429 DAILY_LIMIT
    └─ 未超 → 放行
```

| 方案 | 月价 | 年价 | 可访问专家 | 日限额 | AI 回复深度 |
|------|------|------|-----------|--------|------------|
| 试用 | 免费 | — | 全部（3 条） | 30/天 | 512 tokens |
| Start | $9.9 | $99.9 | Evan, Liam | 30/天 | 512 tokens |
| Pro | $19.9 | $199.9 | 全部 | 100/天 | 1024 tokens |
| Ultra | $39.9 | $399.9 | 全部 | 10000/天 | 2048 tokens |

## 数据统计引擎

```
analytics_events (埋点写入)
    ↓ Vercel Cron Job 定时聚合
analytics_daily_stats (日级汇总: dau/pv/uv/message_count/payment_total 等)
    ↓ 月末汇总
analytics_monthly_stats (月级汇总)

analytics_retention (留存率独立计算: cohort_date + dayN)
```

查询接口全部在 `lib/stats/query.ts` 中，使用原生 SQL（`db.execute(sql...)`）保证查询灵活性。

## 部署

### 流程

```
本地开发 → git push origin master (SSH: git@github.com:blacker135/qfriendly.git)
    ↓
GitHub webhook → Vercel 自动拉取 → 构建 + 部署
```

### 环境变量管理

| 文件 | 用途 |
|------|------|
| `.env.local` | 本地开发（不提交 Git） |
| `.env.vercel.prod` | Vercel 生产环境快照（`vercel env pull production`） |
| `.env.vercel.dev` | Vercel 预览环境快照（`vercel env pull preview`） |
| `.env.local.example` | 模板（提交 Git） |

同步规则: Vercel 和 `.env.local` 保持一致。Vercel Dashboard 修改后，`vercel env pull` 更新 `.env.vercel.*` 文件。

### 部署配置

vercel.json 配置定时任务，API 区域在每个 route.ts 顶层通过 `export const regions = ['hkg1']` 指定。

```json
// vercel.json
{ "crons": [{ "path": "/api/cron/cleanup", "schedule": "0 3 * * *" }] }
```

所有 API 函数部署在香港区域以最小化延迟。
